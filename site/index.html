The idea

Who can read the branch is who can read its secrets.

Grant read on staging and that person can decrypt staging's secrets — automatically. One list, not two. Revoke them and the secrets re-encrypt without them.

01 / MARK

Mark what's secret

List the secret paths in .gitattributes (.env, *.pem, secrets/**). It's committed, so the rule travels with the repo.

02 / ADD

Encrypted on add

When you git add, gitsafe finds who can read the current branch and encrypts the file to exactly them with age. Git stores the encrypted version.

03 / CHECKOUT

Decrypted on checkout

If you're allowed, the file is decrypted for you. If not, you get a clear "locked" note instead of the secret — your checkout never breaks.

Is it safe?

Checked before trusted.

gitsafe won't encrypt anything against an access list it can't verify — so nobody can trick you into encrypting a secret to their own key.

The access list can't be forged

It's signed and stored in the repo, and each clone pins its fingerprint the first time you trust it (like SSH). Tamper with it or swap it out, and gitsafe stops.

Private keys stay out — and honest limits

Keys live in ~/.config/gitsafe, never in the repo (optionally passphrase-locked). The threat model spells out exactly what's protected and what isn't.

Quickstart

Protect your first secret in four lines.

$ gitsafe key gen                 # one time: create your private key
$ gitsafe init --user alice       # turn gitsafe on for this repo
$ echo "API_KEY=sk-live-…" > .env
$ git add .env && git commit -m "add a secret"

# give a teammate access to one branch — add, grant, and rotate in one step:
$ gitsafe onboard bob staging --sign <hex> --enc age1…  # bob can now read staging
